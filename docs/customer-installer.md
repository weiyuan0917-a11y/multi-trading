# MultiTrading Customer Installer

Customer build goals:

- Customers can install and start MultiTrading without installing Python, Node.js, npm, or frontend dependencies.
- The package does not ship frontend source directories such as `frontend/app`, `frontend/components`, `frontend/lib`, or `frontend/convex`.
- Customer navigation keeps the subscription/upgrade page, but does not expose payment-order administration or License issuing pages.
- Customer Next.js API routes for `/api/admin/*` return `not_available_in_customer_build`; `/api/billing/manual-orders` remains available so customers can create payment orders.
- Local runtime data stays under the installed `data/` and `logs/` folders.

## Build

From the repository root:

```powershell
.\build_customer_installer.bat -Version 1.0.0
```

Fast rebuild when `dist/Backend.exe` already exists:

```powershell
.\build_customer_installer.bat -Version 1.0.0 -ReuseBackend
```

Fast rebuild when both `dist/Backend.exe` and `frontend/.next/standalone/server.js` already exist:

```powershell
.\build_customer_installer.bat -Version 1.0.0 -ReuseBackend -ReuseFrontend
```

The script creates:

- `dist/customer/MultiTrading/`: source-less customer runtime folder.
- `dist/customer/MultiTradingSetup-<version>.exe`: one-click self-contained installer generated by Go.
- `dist/customer/MultiTradingCustomer.iss`: optional Inno Setup script for a traditional installer.

## Runtime

The installed launcher starts:

- backend: `127.0.0.1:8010`
- frontend: `127.0.0.1:3010`

Customer first-run flow:

1. Log in with a local account.
2. Configure broker, market data, Feishu, and other local settings in Setup.
3. Import the administrator-issued local License in Personal Center.

## Customer Data Sources

The customer backend bundles the CN/A-share public data-source packages used by the Setup page:

- `mootdx`
- `akshare`
- `baostock`
- `tushare`

The build script temporarily installs the `mootdx` dependency set before creating `Backend.exe`, then restores the developer environment pins used by MCP / google-genai tooling. This keeps the customer package simple while avoiding dependency churn in the source checkout.

## Customer Subscription Page

The customer build includes `/billing` for subscription upgrades. Configure this in the packaged runtime `.env` or build environment if customer machines should create payment orders directly:

```text
BILLING_PUBLIC_ORDER_API_URL=https://your-admin-domain.example/api/billing/manual-orders
```

Recommended flow:

1. Customer opens `/billing` and clicks "Generate payment order".
2. The customer local Next route posts to `BILLING_PUBLIC_ORDER_API_URL`.
3. The administrator deployment creates a pending row in the shared `manualOrders` table.
4. The administrator opens `/admin/orders`, confirms payment, and issues the License.

For builds where no administrator Next deployment is available, the local route can also post to a Convex public HTTP action:

```text
BILLING_PUBLIC_ORDER_API_URL=https://<your-convex-site>/billing/public/manual-orders
```

That Convex route only creates pending payment orders. Listing orders, confirming payment, and issuing License remain protected by the administrator webhook/admin routes.

If `BILLING_PUBLIC_ORDER_API_URL` is not set, the local route falls back to the administrator Convex webhook configuration, which requires `CONVEX_SITE_URL` and `MT_BILLING_WEBHOOK_SECRET`. Do not distribute administrator webhook secrets to customer machines.

## Admin Modules

The customer build keeps this customer page:

- `/billing`

The customer build hides these administrator pages:

- `/admin/orders`
- `/admin/licenses`

The customer build also blocks the related administrator Next.js API routes. Payment order administration and License issuing stay in the administrator deployment/source environment and should not be distributed with customer installers.
